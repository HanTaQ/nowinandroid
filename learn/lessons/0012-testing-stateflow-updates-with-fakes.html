第 12 课：用 Fake 驱动 StateFlow 更新

上一课只读取了 stateIn 的初始值。本课让测试依赖主动发出数据，观察两个 UI State 如何产生不同的新状态。

第 11 课反馈

四题全部正确。你已经能识别本地 ViewModel 测试、测试依赖、主线程 dispatcher 的替换，以及“读取初始值”和“收集上游更新”的差别。

本课只读这一条测试

@Test
fun onboardingStateIsLoadingWhenTopicsAreLoading() = runTest {
    backgroundScope.launch(UnconfinedTestDispatcher()) {
        viewModel.onboardingUiState.collect()
    }
    backgroundScope.launch(UnconfinedTestDispatcher()) {
        viewModel.feedState.collect()
    }

    userDataRepository.setFollowedTopicIds(emptySet())

    assertEquals(
        OnboardingUiState.Loading,
        viewModel.onboardingUiState.value,
    )
    assertEquals(
        NewsFeedUiState.Success(emptyList()),
        viewModel.feedState.value,
    )
}

1. 先按 Arrange、Act、Assert 分段

Arrange

启动两个 collector，让两个使用 WhileSubscribed 的 StateFlow 开始收集各自的上游。

Act

调用 Fake Repository 的测试 API，发出“用户没有关注任何 topic”的数据。

Assert

读取两个 StateFlow.value，验证当前 UI 状态是否符合预期。

2. 为什么要启动两个 collector

onboardingUiState 和 feedState 是两个独立调用 stateIn(...WhileSubscribed...) 创建的 StateFlow。订阅其中一个，不会自动订阅另一个。

collector A ──订阅──> onboardingUiState ──收集──> combine(...)

collector B ──订阅──> feedState       ──收集──> repository Flow

如果删除 feedState.collect()，那么 feedState 不会开始收集 repository 的新值，读取 feedState.value 时通常仍会得到它的初始值 Loading。

3. backgroundScope 和 UnconfinedTestDispatcher

collect() 会持续等待后续数据，正常情况下不会自行结束。把它放进 backgroundScope，测试主体可以继续执行；测试结束时，这个后台 collector 会被自动取消。

UnconfinedTestDispatcher 让新启动的 collector 立即进入执行，确保下一行 Fake Repository 发出数据之前，订阅已经建立。这里的目的不是模拟真实线程速度，而是让测试执行顺序确定。

4. Fake Repository 怎样制造输入

TestUserDataRepository 内部使用可由测试控制的 MutableSharedFlow<UserData>：

private val _userData = MutableSharedFlow<UserData>(
    replay = 1,
    onBufferOverflow = DROP_OLDEST,
)

suspend fun setFollowedTopicIds(followedTopicIds: Set<String>) {
    _userData.tryEmit(
        currentUserData.copy(followedTopics = followedTopicIds),
    )
}

生产环境的 Repository 会读 DataStore、Room 或网络；测试中的 Fake 保持相同接口，却允许测试直接指定下一份数据。这就是 dependency injection 提高可测试性的具体收益。

5. 为什么两个最终状态不同

测试只向 userDataRepository 发出了数据，没有向 topicsRepository 发出 topics。

feedState 路径

followedTopics = emptySet()
  → observeAllForFollowedTopics()
  → 没有关注的 topic，直接产生 emptyList()
  → map(NewsFeedUiState::Success)
  → Success(emptyList())

onboardingUiState 路径

userData 已经发出
topicsRepository 仍未发出 topics
  → combine(shouldShowOnboarding, getFollowableTopics())
    还缺少第二个上游的首个值
  → 没有产生新的 OnboardingUiState
  → 保持 initialValue = Loading
关键结论

combine 必须等每个上游都至少发出一个值，才能执行 transformation；但 feedState 的链路不依赖 topicsRepository，所以它已经可以完成。
