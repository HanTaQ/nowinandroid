第 3 课：TopicSelection 的状态、身份与事件

这一课只读一个 composable：TopicSelection。目标是把 rememberLazyGridState、items(key)、onClick 放到同一个思维模型里。

这段代码在做什么

@Composable
private fun TopicSelection(
    onboardingUiState: OnboardingUiState.Shown,
    onTopicCheckedChanged: (String, Boolean) -> Unit,
    modifier: Modifier = Modifier,
) {
    val lazyGridState = rememberLazyGridState()

    LazyHorizontalGrid(
        state = lazyGridState,
        rows = GridCells.Fixed(3),
    ) {
        items(
            items = onboardingUiState.topics,
            key = { it.topic.id },
        ) {
            SingleTopicButton(
                name = it.topic.name,
                topicId = it.topic.id,
                isSelected = it.isFollowed,
                onClick = onTopicCheckedChanged,
            )
        }
    }
}

1. rememberLazyGridState：保留滚动状态

rememberLazyGridState() 创建并记住一个 LazyGridState。这个 state 主要保存 lazy grid 的滚动位置、滚动偏移等 UI 层临时状态。

读法

这个 state 属于“屏幕交互状态”，不是业务状态。它不用进 ViewModel，因为它不代表用户关注了哪些 topic，只代表这个网格当前滚到哪里。

如果不 remember，每次 recomposition 都可能重新创建 state，滚动位置就容易被重置。用 remember 后，同一个 composable 位置在重组时会继续使用之前的 state 对象。

2. items(key)：给列表 item 一个稳定身份

key = { it.topic.id } 的重点不是 RecyclerView 那种 ViewHolder 回收，而是告诉 Compose：每个 topic item 的稳定身份是 topic.id。

没有 key

按位置猜

如果列表从 A, B, C 变成 X, A, B, C，Compose 更容易按位置理解变化：第 0、1、2 项都变了。

有 key

按身份对齐

Compose 知道 A 还是 A，B 还是 B，只是位置移动了。这样能更好地保留对应 item 的 composition 和内部状态。

在这个项目里，topic 的名字、图片、是否关注都可能变化；用 topic.id 做 key，可以让 UI 更新时更稳定地把“旧 item 的身份”对应到“新列表中的同一个 topic”。

3. onClick：组件不私自改业务状态

SingleTopicButton 只接收当前是否选中：

isSelected = it.isFollowed

点击时，它不自己保存“我现在选中了”。它只是把事件交出去：

onClick(topicId, !isSelected)

这个事件一路回到 ForYouViewModel.updateTopicSelection(topicId, isChecked)，再由 ViewModel 写入 repository。之后 repository / use case 发出新的数据，onboardingUiState 更新，UI 再拿到新的 isFollowed。

把三者连起来

  1. rememberLazyGridState 保存 UI 交互状态：网格滚动到哪里。
  2. items(key) 保存列表身份关系：哪个 item 对应哪个 topic。
  3. onClick 把业务事件交给上层：topic 是否关注由数据层回流决定。
