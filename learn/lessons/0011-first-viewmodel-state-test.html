第 11 课：读懂第一条 ViewModel 状态测试

你已经追完了 feed 状态进入 UI 的路径。现在从最短的一条测试开始，验证 ViewModel 是否向 UI 提供了正确的初始状态。

先补清第 10 课第 4 题

ForYouScreen 的主体是一个 Box。Box 可以把多个并列子项放在同一块区域中，后声明的子项可以画在前面子项的上方：

Box {
    LazyVerticalStaggeredGrid { ... } // 第 1 层：页面列表内容
    AnimatedVisibility { ... }       // 第 2 层：顶部加载提示
    DraggableScrollbar(...)          // 第 3 层：滚动条
}

因此，所谓“屏幕层”并不是一个 Compose API 名称，而是描述这段 UI 的职责：加载提示由整个 ForYouScreen 统一绘制，不属于 newsFeed 这个列表 section。

val isFeedLoading = feedState is NewsFeedUiState.Loading

AnimatedVisibility(
    visible = isSyncing || isFeedLoading || isOnboardingLoading,
) {
    NiaOverlayLoadingWheel(...)
}

|| 是逻辑“或”。三个条件只要有一个是 true，visible 就是 true，加载提示会通过进入动画显示；全部为 false 时，它通过退出动画隐藏。

所以当 feedState 是 Loading 时，会同时发生两件事：

  1. newsFeed 的 Loading -> Unit 不向 Lazy grid 添加新闻卡片。
  2. isFeedLoading 为 true，外层 AnimatedVisibility 显示 NiaOverlayLoadingWheel。
一句话模型

newsFeed 决定“列表里有没有新闻卡片”，外层 AnimatedVisibility 决定“页面上方有没有加载提示”。

本课目标：读懂最短的 ViewModel 测试

打开 feature/foryou/impl/src/test/.../ForYouViewModelTest.kt，先只看这一条：

@Test
fun stateIsInitiallyLoading() = runTest {
    assertEquals(
        OnboardingUiState.Loading,
        viewModel.onboardingUiState.value,
    )
    assertEquals(
        NewsFeedUiState.Loading,
        viewModel.feedState.value,
    )
}

1. 这条测试验证什么

它验证 ForYouViewModel 刚创建、repository 还没有提供数据时，onboarding 和 feed 都处于 Loading。这正是第 10 课在 UI 中使用的两个加载条件。

测试名称可以按行为读成：

state Is Initially Loading
状态  在初始时    是加载中

2. ViewModel 在哪里创建

@Before 标记的方法会在每条测试运行前执行：

@Before
fun setup() {
    viewModel = ForYouViewModel(
        syncManager = syncManager,
        savedStateHandle = savedStateHandle,
        analyticsHelper = analyticsHelper,
        userDataRepository = userDataRepository,
        userNewsResourceRepository = userNewsResourceRepository,
        getFollowableTopics = getFollowableTopicsUseCase,
    )
}

这里注入的是测试依赖。测试不会启动真实 App，也不会要求用户点击 UI；它直接创建被测的 ForYouViewModel。

3. 为什么需要 MainDispatcherRule

@get:Rule
val mainDispatcherRule = MainDispatcherRule()

viewModelScope 默认依赖主线程 dispatcher，但本地 JVM 测试没有 Android 主线程。这个 rule 在测试开始时用 TestDispatcher 替换 Dispatchers.Main，测试结束后再恢复。

4. runTest 提供什么

runTest 为协程测试提供 TestScope 和可控制的调度环境。后面的测试会在这里启动 collector、调用 suspend 函数，并控制尚未完成的异步任务。本条测试虽然只读取初始值，但沿用相同测试结构。

5. 为什么这条测试没有调用 collect()

stateIn(..., initialValue = Loading) 创建 StateFlow 时，初始值已经可以通过 .value 立即读取。这条测试只验证初始值，不要求它从 repository 接收新数据，因此不需要启动 collector。

下一条测试若要验证 repository 发出数据后的状态，就不同了。由于代码使用 SharingStarted.WhileSubscribed，测试期间必须至少存在一个 collector，stateIn 才会开始收集上游：

backgroundScope.launch(UnconfinedTestDispatcher()) {
    viewModel.feedState.collect()
}

先记住区别即可：

初始值

只检查 initialValue：直接断言 StateFlow.value。

上游更新

要接收 repository 后续 emission：先启动 collector，再制造输入并断言。
