第 5 课：UserDataRepository 的读流与写命令

这一课读懂 UserDataRepository 的 API 形状：为什么它同时有 val userData: Flow<UserData> 和一组 suspend fun set...。

先校正一个 Kotlin 点

interface UserDataRepository {
    val userData: Flow<UserData>
    suspend fun setTopicIdFollowed(followedTopicId: String, followed: Boolean)
}

val userData 在 Kotlin interface 里不是“接口里存了一个变量”。它声明的是一个只读属性契约：任何实现 UserDataRepository 的类，都必须提供一个可以读取的 userData 属性。

Kotlin

属性是访问器契约

val userData 可以理解成要求实现类提供一个 getter。它不等于接口自己持有字段。

Java 对比

更像 getUserData()

如果用 Java 思维理解，它接近一个 Flow<UserData> getUserData() 方法，而不是接口里的普通成员变量。

读：持续观察 UserData

val userData: Flow<UserData>

这表示 user data 是一个会随时间变化的数据流。调用者不是“拿一次当前值就完了”，而是订阅它：当 followed topics、bookmarks、theme、onboarding 状态变化时，新的 UserData 会继续发出来。

在 For You 里

ForYouViewModel 通过 userDataRepository.userData.map { !it.shouldHideOnboarding } 得到 shouldShowOnboarding。用户状态变化后，onboarding UI 会自动重新计算。

写：一次性发出修改命令

suspend fun setTopicIdFollowed(
    followedTopicId: String,
    followed: Boolean,
)

写操作是一次性的命令，所以它是 function。它可能访问磁盘 DataStore，也可能做日志记录，因此是 suspend。

在实现类里，它长这样：

override suspend fun setTopicIdFollowed(followedTopicId: String, followed: Boolean) {
    niaPreferencesDataSource.setTopicIdFollowed(followedTopicId, followed)
    analyticsHelper.logTopicFollowToggled(followedTopicId, followed)
}

Repository 不只是转发：它把数据写入 NiaPreferencesDataSource，同时记录 analytics。这就是 repository 作为数据层公共 API 的价值。

真正的 source of truth：DataStore

override val userData: Flow<UserData> =
    niaPreferencesDataSource.userData

OfflineFirstUserDataRepository 的 userData 直接来自 NiaPreferencesDataSource.userData。再往下看，NiaPreferencesDataSource 从 Proto DataStore 读取 UserPreferences，并映射成 app 使用的 UserData。

val userData = userPreferences.data
    .map {
        UserData(
            bookmarkedNewsResources = it.bookmarkedNewsResourceIdsMap.keys,
            viewedNewsResources = it.viewedNewsResourceIdsMap.keys,
            followedTopics = it.followedTopicIdsMap.keys,
            shouldHideOnboarding = it.shouldHideOnboarding,
            ...
        )
    }

这意味着 UI state 不是凭空来的，它最终来自 DataStore 中持久化的用户偏好。

为什么 Topic 里没有 isFollowed

你第 4 课第 2 题已经答对：Topic 是内容数据，来自 topics 表；followedTopics 是用户偏好，来自 user data。它们属于不同数据源。

FollowableTopic(
    topic = topic,
    isFollowed = topic.id in userData.followedTopics,
)

所以 isFollowed 不是 topic 自己的属性，而是 “topic + 当前用户数据” 组合后的结果。这就是 GetFollowableTopicsUseCase 存在的原因。

读写分离的形状

Read

Flow

val userData: Flow<UserData> 用于持续观察状态变化。

Write

suspend fun

setTopicIdFollowed(...) 用于发出一次修改命令。

Loop

写完再读回来

写操作更新 DataStore，DataStore 的 Flow 发出新值，ViewModel 再生成新的 UI state。
