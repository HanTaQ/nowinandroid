第 4 课：一次 topic 点击如何变成新的 UI state

这一课把第 3 课的 onClick 往下追：用户点击 topic 后，事件如何写入 repository，又如何通过 Flow 回到 ForYouScreen。

先看完整路径

SingleTopicButton 触发点击事件

TopicSelection 把事件交给 onTopicCheckedChanged

ForYouScreen 连接到 viewModel::updateTopicSelection

ForYouViewModel 调用 userDataRepository.setTopicIdFollowed(topicId, isChecked)

UserDataRepository.userData 发出新的 UserData

GetFollowableTopicsUseCase 用新的 UserData 重新计算 FollowableTopic

ForYouViewModel.onboardingUiState 重新发出 Shown(topics)

ForYouScreen recomposition，新的 isSelected 传回 SingleTopicButton

1. UI 只发事件

Surface(
    selected = isSelected,
    onClick = {
        onClick(topicId, !isSelected)
    },
)

SingleTopicButton 读 isSelected 来显示当前 UI，但它不把新的 selected 状态存在自己内部。点击发生时，它只说：“这个 topic 想切到另一个 checked 状态”。

2. ViewModel 处理事件

fun updateTopicSelection(topicId: String, isChecked: Boolean) {
    viewModelScope.launch {
        userDataRepository.setTopicIdFollowed(topicId, isChecked)
    }
}

这里有两个值得注意的点：

  • 写入 repository 是 suspend 操作，所以放进 viewModelScope.launch。
  • ViewModel 没有直接改某个 Compose state，而是改数据源，让数据流自己回到 UI。

3. Repository 暴露 userData Flow

interface UserDataRepository {
    val userData: Flow<UserData>
    suspend fun setTopicIdFollowed(followedTopicId: String, followed: Boolean)
}

这个接口同时提供 read stream 和 write function。读是持续的 Flow<UserData>，写是一次性的 suspend function。Now in Android 很多数据层接口都采用这种形状。

4. Use case 把 UserData 和 Topics 合成 FollowableTopic

operator fun invoke(): Flow<List<FollowableTopic>> = combine(
    userDataRepository.userData,
    topicsRepository.getTopics(),
) { userData, topics ->
    topics.map { topic ->
        FollowableTopic(
            topic = topic,
            isFollowed = topic.id in userData.followedTopics,
        )
    }
}

这一步非常关键：topic 本身来自 topics repository；“是否关注”来自 user data。Use case 把两者组合成 UI 更容易使用的 FollowableTopic。

5. ViewModel 再合成 OnboardingUiState

combine(
    shouldShowOnboarding,
    getFollowableTopics(),
) { shouldShowOnboarding, topics ->
    if (shouldShowOnboarding) {
        OnboardingUiState.Shown(topics = topics)
    } else {
        OnboardingUiState.NotShown
    }
}

注意这里出现了第二层 combine。第一层 combine 在 use case 里，负责合成 FollowableTopic。第二层 combine 在 ViewModel 里，负责合成页面要渲染的 OnboardingUiState。

Use case

业务形状

Topic + UserData -> FollowableTopic

ViewModel

页面形状

shouldShow + topics -> OnboardingUiState

Compose

渲染形状

OnboardingUiState.Shown -> TopicSelection

这就是 UDF

这一整条链就是 Now in Android 中的 unidirectional data flow：

  • 事件向下：UI -> ViewModel -> Repository/DataStore
  • 数据向上：DataStore/Repository -> UseCase -> ViewModel UI state -> Compose UI

所以第 3 课你说“事件交给 ViewModel，新的 UI 状态再从数据来源回来”是对的。第 4 课只是把这句话拆成可追踪的源码路径。
