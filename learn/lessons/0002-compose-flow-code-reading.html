第 2 课：读懂两个卡点

这一课先解决两个真实源码片段：heightIn(max = max(...)) 和 combine(...)。它们分别代表 Compose layout 体系和 Kotlin Flow 状态组合体系。

卡点一：LazyHorizontalGrid 为什么要 heightIn

源码位置：feature/foryou/impl/.../ForYouScreen.kt 的 TopicSelection。

Modifier
    .heightIn(max = max(240.dp, with(LocalDensity.current) { 240.sp.toDp() }))
    .fillMaxWidth()
    .testTag(topicSelectionTestTag)

先把这行翻译成人话：这个横向网格的高度不要超过“240dp”和“按当前字体缩放后的 240sp”两者中更大的那个。

为什么要限制高度

LazyHorizontalGrid 是横向滚动网格。它需要知道自己在垂直方向最多能占多少空间，否则内部三行 item 无法稳定测量。

为什么不是固定 240dp

用户可能把系统字体调大。topic 文本会变高。如果固定 240dp，大字体下文字可能放不下。

为什么用 240sp.toDp()

sp 会受字体缩放影响。把 240sp 转成 dp 后，就得到“当前字体设置下，需要给文字预留的布局高度”。

所以这行代码的意图是：普通字体时最多 240dp；大字体时最多给到更高的空间。它是在为 accessibility 和 adaptive UI 留余量。

卡点二：combine 怎么生成 onboardingUiState

源码位置：feature/foryou/impl/.../ForYouViewModel.kt。

combine(
    shouldShowOnboarding,
    getFollowableTopics(),
) { shouldShowOnboarding, topics ->
    if (shouldShowOnboarding) {
        OnboardingUiState.Shown(topics = topics)
    } else {
        OnboardingUiState.NotShown
    }
}

这段代码的读法是：我有两个会随时间变化的数据来源，一个告诉我“要不要显示 onboarding”，另一个告诉我“可关注 topic 列表是什么”。只要其中任意一个更新，就用两个来源的最新值重新算出一个 OnboardingUiState。

来源 A

shouldShowOnboarding

来自用户数据。用户没有关闭过 onboarding 时，它是 true。

来源 B

getFollowableTopics()

来自 use case。它把 topic 数据和用户已关注状态合成 List<FollowableTopic>。

输出

OnboardingUiState

如果需要显示，就输出 Shown(topics)；否则输出 NotShown。

后面接的 stateIn(..., initialValue = OnboardingUiState.Loading) 是为了把这个 Flow 变成 UI 可以稳定收集的 StateFlow。在两个来源还没都发出值之前，UI 先看到 Loading。

新的学习安排

  1. 每读一个 feature screen，先标出三个东西：输入 state、输出 event、内部临时 UI state。
  2. 遇到 Modifier，先问：它在改尺寸、位置、外观、点击行为，还是测试语义？
  3. 遇到 Flow 链，先问：数据来源是什么、何时重新发出、最后变成什么 UI state？
  4. 遇到 stateIn，先看三个参数：scope、started、initialValue。
