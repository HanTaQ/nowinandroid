第 9 课：NewsResourceQuery 如何变成新闻列表

这一课只解决一个卡点：NewsResourceQuery(filterTopicIds = followedTopics) 到底做了什么。

先反馈你的第 8 课练习

  • 第 1 题正确：distinctUntilChanged() 过滤的是“连续相同”的 emission，目的是避免下游重复查询和重复计算。
  • 第 2 题正确：flatMapLatest 会根据最新的 followedTopics 创建新的内部 Flow，并取消旧的内部 Flow。
  • 第 4 题方向正确：UserNewsResource 不只是新闻内容，还带着当前用户相关状态，例如是否收藏、是否看过、topic 是否已关注。
  • 第 3 题卡住的地方很关键：NewsResourceQuery 不是数据库查询本身，它只是 data 层传递查询条件的对象。

1. NewsResourceQuery 是“查询条件对象”

源码在 core/data/.../NewsRepository.kt：

data class NewsResourceQuery(
    val filterTopicIds: Set<String>? = null,
    val filterNewsIds: Set<String>? = null,
)

可以把它读成：

  • filterTopicIds = null：不按 topic 过滤，任何 topic 都可以。
  • filterTopicIds = setOf("compose")：只要和 compose 这个 topic 相关的新闻。
  • filterNewsIds = null：不按新闻 id 过滤，任何新闻 id 都可以。
  • filterNewsIds = setOf("news-1")：只要指定 id 的新闻。
关键区别

null 表示“不启用这个过滤条件”；空集合 emptySet() 表示“启用过滤，但没有任何 id 能匹配”。所以这里不能简单把 null 和空集合当成一回事。

2. followedTopics 变成 query

CompositeUserNewsResourceRepository 里这段代码的意思是：用户关注了哪些 topic，就用这些 topic 作为新闻查询条件。

userDataRepository.userData
    .map { it.followedTopics }
    .distinctUntilChanged()
    .flatMapLatest { followedTopics ->
        when {
            followedTopics.isEmpty() -> flowOf(emptyList())
            else -> observeAll(
                NewsResourceQuery(filterTopicIds = followedTopics)
            )
        }
    }

如果用户关注了 {"compose", "kotlin"}，这里就会创建：

NewsResourceQuery(
    filterTopicIds = setOf("compose", "kotlin"),
    filterNewsIds = null,
)

到这一步，还没有真正执行 SQL。它只是把“我要按这些 topic 查新闻”表达出来。

3. observeAll 把 query 交给 NewsRepository

override fun observeAll(
    query: NewsResourceQuery,
): Flow<List<UserNewsResource>> =
    newsRepository.getNewsResources(query)
        .combine(userDataRepository.userData) { newsResources, userData ->
            newsResources.mapToUserNewsResources(userData)
        }

这里分两步：

  1. newsRepository.getNewsResources(query)：按 query 拿到 Flow<List<NewsResource>>。
  2. combine(userDataRepository.userData)：再把新闻内容和用户状态合成 UserNewsResource。

4. OfflineFirstNewsRepository 把 query 翻译成 DAO 参数

真正把 NewsResourceQuery 拆开的地方在 OfflineFirstNewsRepository：

newsResourceDao.getNewsResources(
    useFilterTopicIds = query.filterTopicIds != null,
    filterTopicIds = query.filterTopicIds ?: emptySet(),
    useFilterNewsIds = query.filterNewsIds != null,
    filterNewsIds = query.filterNewsIds ?: emptySet(),
)

读法是：

  • 如果 filterTopicIds != null，就启用 topic 过滤。
  • 如果 filterTopicIds == null，就不启用 topic 过滤。
  • 传给 DAO 的集合不能是 null，所以 null 时用 emptySet() 占位。

5. DAO 才是真正的数据库查询

NewsResourceDao.getNewsResources 用 Room SQL 查询本地数据库：

CASE WHEN :useFilterTopicIds
    THEN id IN (
        SELECT news_resource_id FROM news_resources_topics
        WHERE topic_id IN (:filterTopicIds)
    )
    ELSE 1
END

这段 SQL 的意思是：

  • 如果 useFilterTopicIds = true，就只查和这些 topic id 关联的 news resource。
  • 如果 useFilterTopicIds = false，ELSE 1 等价于“这一项条件永远通过”，也就是不按 topic 过滤。

6. 完整类型链

Flow<UserData>
    -> map { it.followedTopics }
Flow<Set<String>>
    -> flatMapLatest { followedTopics ->
           observeAll(NewsResourceQuery(filterTopicIds = followedTopics))
       }
NewsResourceQuery
    -> newsRepository.getNewsResources(query)
Flow<List<NewsResource>>
    -> combine(userDataRepository.userData)
Flow<List<UserNewsResource>>

一句话总结：NewsResourceQuery 是 data 层的过滤条件对象，OfflineFirstNewsRepository 把它翻译成 DAO 参数，NewsResourceDao 才真正查询数据库，最后 CompositeUserNewsResourceRepository 再把新闻内容和用户状态合成 UI 更好用的模型。
