ViewModel 与 Flow 测试速查

阅读 Now in Android 测试代码时，先判断测试位置、协程环境、输入控制方式和断言对象。

四个核心部件

位置

src/test

运行在本地 JVM，适合 ViewModel、use case、repository 等不依赖真实 Android UI 的逻辑。

环境

runTest

提供测试协程作用域、虚拟时间和异步任务调度能力，让挂起函数与 Flow 测试可控。

主线程

MainDispatcherRule

测试开始时替换 Dispatchers.Main，结束后恢复，使依赖 viewModelScope 的代码能在本地测试运行。

断言

stateFlow.value

当目标是验证当前 UI 状态时，直接断言 StateFlow.value，不必把每次 emission 都收集成列表。

stateIn(WhileSubscribed) 的测试规则

// 只检查 initialValue：可以直接读取 value
assertEquals(Loading, viewModel.feedState.value)

// 要让 stateIn 开始收集上游：测试期间要有 collector
backgroundScope.launch(UnconfinedTestDispatcher()) {
    viewModel.feedState.collect()
}

fakeRepository.emit(...)
assertEquals(expected, viewModel.feedState.value)
为什么

SharingStarted.WhileSubscribed 的意思就是“有订阅者时才收集上游”。读取 value 只能读当前缓存值，它本身不算订阅。

测试 collector 的两个常见部件

生命周期

backgroundScope

适合运行不会自行结束的 collect()；测试结束时，后台协程会被自动取消。

启动顺序

UnconfinedTestDispatcher

让 collector 立即开始执行，避免 Fake 发出数据时订阅还没有建立。

多个 stateIn 要分别订阅

viewModel.onboardingUiState.collect() // 启动 onboarding 上游
viewModel.feedState.collect()         // 启动 feed 上游

两个 StateFlow 各自使用 SharingStarted.WhileSubscribed 时，订阅其中一个不会启动另一个。

一条常见测试路径

Arrange：创建 fake 依赖和 ViewModel
   ↓
Collect：需要上游更新时，启动 StateFlow collector
   ↓
Act：让 fake repository 发出测试数据，或调用 ViewModel 事件
   ↓
Assert：读取 StateFlow.value，与期望 UI state 比较

多上游状态的检查顺序

1. 给每个 Flow 标出元素类型
2. 找出 map / combine 后的输出类型
3. 确认每个 combine 的上游是否都已发出首个值
4. 确认目标 stateIn 是否有 collector
5. 最后预测 StateFlow.value

combine 只有在每个上游至少发出一个值后，才能执行 transformation。测试某个状态仍为 Loading 时，优先检查是否缺少上游首值或 collector。
