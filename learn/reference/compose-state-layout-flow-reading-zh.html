1. Compose State：UI 是 state 的函数

Compose 是 declarative UI。你不直接命令某个 View 改文字、改颜色、隐藏或显示；你把新的 state 传给 composable，Compose 重新执行相关 composable，也就是 recomposition，然后根据新的 state 生成新的 UI。

核心句

state 变，UI 重新计算

Composable 要更新，必须显式收到新的 state。普通可变对象如果不是 Compose 可观察的 state，改了也不一定触发 recomposition。

Now in Android

ViewModel 暴露 StateFlow

ForYouViewModel 暴露 onboardingUiState、feedState、isSyncing。这些是 UI 可以订阅的状态来源。

Compose 入口

collectAsStateWithLifecycle

ForYouScreen 使用 collectAsStateWithLifecycle() 把 Flow/StateFlow 转成 Compose 可观察的 state，并且会跟随 Android lifecycle 安全收集。

读源码口诀

看到 val x by viewModel.xxx.collectAsStateWithLifecycle()，就读成：UI 正在订阅 ViewModel 的某个状态；这个状态一变化，相关 UI 会重新组合。

2. State hoisting：状态下传，事件上传

State hoisting 是 Compose 中让组件更可复用、更容易测试的常见模式。它把组件内部状态移到调用者那里，让组件尽量 stateless。

value: T
onValueChange: (T) -> Unit

在 Now in Android 中，它常常长这样：

SingleTopicButton(
    isSelected = it.isFollowed,
    onClick = onTopicCheckedChanged,
)

SingleTopicButton 不自己决定 topic 是否被关注。它只显示 isSelected，用户点击后把事件交给外层。外层最终调用 ViewModel，ViewModel 更新 repository，repository 的 Flow 再发出新数据，UI 收到新 state 后重新显示。

3. Compose 布局约束：父组件给范围，子组件在范围内测量

Compose layout 可以粗略理解为三步：先测量 children，再决定自己的 size，最后放置 children。父组件测量子组件时，会给子组件一组 constraints，也就是最小/最大宽高范围。

Constraints

不是想多大就多大

子组件只能在父组件传下来的 min/max width/height 范围内选择自己的尺寸。

Modifier

会改约束

有些 modifier 只改外观或语义；有些 modifier 会改变测量约束，例如 size、heightIn、widthIn、fillMaxWidth。

顺序

链式顺序有意义

Modifier 是一层层包住 layout node。前面的 modifier 会影响后面的测量、绘制或输入行为，所以顺序不是随便写的。

回到 ForYouScreen.kt

Modifier
    .heightIn(max = max(240.dp, with(LocalDensity.current) { 240.sp.toDp() }))
    .fillMaxWidth()
    .testTag(topicSelectionTestTag)

这段 modifier 的读法：

  • heightIn(max = ...)：给这个 LazyHorizontalGrid 的高度设置上限。
  • fillMaxWidth()：宽度尽量占满父组件允许的最大宽度。
  • testTag(...)：给 UI 测试提供查找节点的标签，不影响布局尺寸。

LazyHorizontalGrid 是横向滚动的 lazy grid。它需要在垂直方向有明确的高度边界，尤其这里还设置了 rows = GridCells.Fixed(3)，也就是固定三行。没有稳定高度，内部 item 的测量会很难成立。

4. Dp、Sp 与 LocalDensity：为什么要 240.sp.toDp()

dp 通常用于布局尺寸，sp 通常用于文字尺寸，并且会受到用户系统字体缩放影响。Compose 中如果要把 sp 换算成 dp，需要当前环境的 density 和 fontScale，这个环境由 LocalDensity.current 提供。

with(LocalDensity.current) {
    240.sp.toDp()
}

这里的意思是：在当前屏幕密度和字体缩放环境中，把 240sp 转成布局可用的 dp。240.sp 是要转换的值，LocalDensity.current 是转换所需的环境。

所以这句：

max(240.dp, with(LocalDensity.current) { 240.sp.toDp() })

可以读成：高度上限至少是 240dp；如果用户把字体调大，那就按字体缩放后的高度给更多空间。这是在照顾 accessibility。

5. Flow combine：多个来源，合成一个 UI state

combine 的作用是把多个 Flow 的最新值拿出来，通过一个 transform 函数合成新的 Flow。它通常要等每个上游 Flow 至少发出一个值，才有足够的“最新值”进行第一次合成。之后任意一个上游更新，都会重新执行 transform。

combine(
    shouldShowOnboarding,
    getFollowableTopics(),
) { shouldShowOnboarding, topics ->
    if (shouldShowOnboarding) {
        OnboardingUiState.Shown(topics = topics)
    } else {
        OnboardingUiState.NotShown
    }
}
Flow A

shouldShowOnboarding

来自用户数据。它表达“用户是否还应该看到 onboarding”。

Flow B

getFollowableTopics()

来自 use case。它表达“当前可关注 topic 列表，以及每个 topic 是否已关注”。

输出 Flow

OnboardingUiState

ViewModel 不把两个 Flow 原样丢给 UI，而是合成 UI 更容易渲染的 sealed state。

这就是 Now in Android UI layer 的典型写法：ViewModel 把领域数据转换成页面状态，Compose 根据页面状态渲染。

6. stateIn：把 cold Flow 变成 UI 可读的 StateFlow

Flow 默认常常是 cold 的：有人 collect 时才开始工作。UI 层更希望拿到一个“随时能读当前值”的状态容器，因此 ViewModel 会用 stateIn 把 Flow 转成 StateFlow。

.stateIn(
    scope = viewModelScope,
    started = SharingStarted.WhileSubscribed(5_000),
    initialValue = OnboardingUiState.Loading,
)
  • scope：在哪个协程作用域里共享这个状态。ViewModel 中通常用 viewModelScope。
  • started：什么时候开始/停止收集上游。WhileSubscribed(5_000) 表示有订阅者时收集，订阅者消失后延迟一段时间再停止。
  • initialValue：在上游还没发出值之前，UI 先看到什么状态。这里是 Loading。

7. 读 Now in Android 源码时怎么用

  1. 看到 collectAsStateWithLifecycle()：找 ViewModel 暴露的 Flow/StateFlow。
  2. 看到 combine：列出每个输入 Flow，再写出输出 UI state。
  3. 看到 stateIn：看初始状态、共享策略、作用域。
  4. 看到 Modifier：逐个判断它影响尺寸、位置、绘制、点击、滚动、语义还是测试。
  5. 看到 heightIn、widthIn、sizeIn：问它在给子组件设置什么约束。
  6. 看到 LocalDensity：通常是在做 dp、sp、px 之间的换算。
